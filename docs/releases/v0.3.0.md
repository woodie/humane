# humane v0.3.0

## `CollapseMinute` renamed to `IncludeSeconds` (breaking)

`TimeFormatter`'s field is now `IncludeSeconds` (zero value `false`), replacing
`CollapseMinute` (recommended value `true`, via `NewTimeFormatter()`).

This isn't just a rename -- the polarity inverts too. `CollapseMinute: true` (via
`NewTimeFormatter()`) suppressed seconds below a minute; `IncludeSeconds: false` (the
zero value) does the exact same thing, so the recommended default behavior is
unchanged. But `CollapseMinute: false` and `IncludeSeconds: true` are the two ends of
that same inversion, not equivalents of each other.

The rename borrows ActionView's own name for this concept -- Rails'
`distance_of_time_in_words` has an `include_seconds:` option that defaults `false`
for the same reason this one does. Reusing the name isn't just convenience: the
defaults independently agree, which is a stronger signal this is the right name than
either language picking its own term would have been. `humane-ruby` picked up the
identical rename in its own `v0.3.0`.

`CollapseMinute` read ambiguously once `humane-swift`'s `approximate` option
(ActionView-inspired "about"/"in about" prefixing on hour-plus buckets) entered the
picture -- "collapse" suggested the same kind of rounding-language operation
`approximate` performs, when this field has only ever been a granularity floor below
a minute. `IncludeSeconds` doesn't compete for that word.

## A side effect: the zero-value gotcha is gone

`TimeFormatter{}`'s zero value used to be the *surprising* case --
`CollapseMinute: false`, second-level granularity, not the recommended default that
`NewTimeFormatter()` returned. That asymmetry was flagged explicitly in the struct
doc comment and README so it wouldn't read as a bug later.

The polarity flip retires it: `IncludeSeconds`' zero value (`false`) *is* the
recommended default now, so `TimeFormatter{}` and `NewTimeFormatter()` produce
identical output. `NewTimeFormatter()` is kept for API stability and parity with the
other two languages' constructors, but it's no longer covering for a Go-specific
footgun -- there's nothing left to trip on. A new test
(`"TimeFormatter{} vs NewTimeFormatter()"`) locks this equivalence in.

## Upgrading

- No explicit `CollapseMinute` field set, or using `NewTimeFormatter()`: no change
  needed, behavior is identical.
- `TimeFormatter{CollapseMinute: false}` (explicit, to get raw seconds): change to
  `TimeFormatter{IncludeSeconds: true}`.
- Bare `TimeFormatter{}` used specifically *to get* the old raw-seconds zero-value
  behavior: that trick no longer works -- the zero value is now the collapsed
  default. Use `TimeFormatter{IncludeSeconds: true}` explicitly instead.

`lambada` is the one known consumer; it needs a follow-up update before pulling this
in, and still hasn't picked up the `v0.2.0` wording change either (see that
release's own upgrade note).
